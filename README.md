# Antifrode Backend (RiskGuard)

Современная антифрод-платформа для анализа транзакций пассажиров, построенная на базе **FastAPI** и принципах **Clean Architecture**. Система позволяет выявлять подозрительную активность, используя комбинацию бизнес-правил и машинного обучения.

---

## 📋 Основные возможности

- **ETL Конвейер**: Загрузка и обработка массивных данных из Excel-файлов, нормализация ФИО и детерминированная идентификация пассажиров.
- **Интеллектуальный Скоринг**: Комбинированный анализ рисков на основе 10+ бизнес-правил и модели аномалий (Isolation Forest).
- **Безопасность**: Полноценная JWT-аутентификация (Access + Refresh токены) с разграничением ролей (Admin/User).
- **Аудит**: Детальное логирование всех значимых действий пользователей.
- **Аналитическая Панель (Dashboard)**: Визуализация трендов риска, концентрации подозрительных операций по каналам, кассам и терминалам.
- **Отчетность**: Генерация сводных отчетов в форматах Excel и DOCX.

## 🚀 Быстрый запуск

### Предварительные требования
* **Docker** и **Docker Compose**
* **Make** (желательно для Windows/Linux)

### Запуск через Docker (рекомендуется)
```bash
# 1. Подготовка окружения (создаст .env из шаблона)
make env-setup

# 2. Запуск всех сервисов (БД + API + ML)
make docker-up

# 3. Сервисы доступны по адресам:
# - Frontend: http://localhost:3000
# - API: http://localhost:8000
# - Документация (Swagger): http://localhost:8000/docs
# - ML Сервис: http://localhost:8001
```

### Первый вход
При первом запуске автоматически создается администратор (если настроен `INITIAL_ADMIN_PASSWORD` в `.env`):
- **Email**: `admin@example.com`
- **Пароль**: `AdminPassword123` (или ваш пароль из .env)

```json
{
  "email": "admin@example.com",
  "password": "AdminPassword123"
}
```

## 🏗 Архитектура проекта

Проект реализован по канонам **Clean Architecture (DDD)**, что обеспечивает высокую тестируемость и независимость бизнес-логики от внешних фреймворков:

1. **Domain Layer (`app/domain/`)**: Сердце системы — сущности, спецификации и интерфейсы репозиториев. Никаких внешних зависимостей.
2. **Application Layer (`app/application/`)**: Варианты использования (Use Cases). Интеракторы связывают доменную логику и инфраструктуру.
3. **Infrastructure Layer (`app/infrastructure/`)**: Техническая реализация — БД, DI, интеграция с ML, конфигурации.
4. **Presentation Layer (`app/presentation/`)**: Внешний слой — маршруты FastAPI, DTO, Middleware и обработка ошибок.

## 📈 ML Модель и Скоринг

Система использует двухэтапный скоринг:
1. **Rule Engine**: Проверка жестких критериев (количество возвратов, ночные покупки, подозрительные ФИО).
2. **Isolation Forest**: Поиск аномальных паттернов поведения, которые сложно описать правилами.

Итоговый риск вычисляется как средневзвешенное значение и классифицируется на группы: **Low**, **Medium**, **High**, **Critical**.

## 🔐 Безопасность

- Пароли хешируются с использованием **bcrypt**.
- Обмен данными защищен **JWT** токенами.
- **Refresh Token** хранится в защищенной `Strict` / `HttpOnly` куке.
- Реализовано Middleware для автоматической проверки аутентификации на защищенных маршрутах.
